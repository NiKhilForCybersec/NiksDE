Compliance Framework Overview

Understanding major security frameworks, their relationships, and how detection engineering supports compliance objectives.

Why Frameworks Matter

Security frameworks provide structured approaches to managing cybersecurity risk. For detection engineers, frameworks help prioritize detection coverage, justify security investments, and demonstrate due diligence to stakeholders.

Risk Management

  • Systematic risk identification
  • Prioritized control implementation
  • Measurable security posture
  • Board-level communication

Regulatory Compliance

  • Meet legal requirements
  • Avoid fines and penalties
  • Enable business operations
  • Maintain certifications

Detection Engineering

  • Coverage gap analysis
  • Detection prioritization
  • Audit evidence generation
  • Control effectiveness metrics

Framework Landscape

Framework Type Primary Use Mandatory?

GDPR Regulation EU data protection Yes (EU data subjects)

Detection Engineering & Compliance

How Detection Supports Controls

Control Area Detection Contribution Example
Access Control Detect unauthorized access attempts Failed login alerts, privilege escalation
Audit & Accountability Log collection, analysis, alerting SIEM rules, log retention policies
Incident Response Detect incidents, enable investigation Malware alerts, compromise indicators
System Integrity Detect unauthorized changes File integrity monitoring, config drift
Data Protection Detect data exfiltration DLP alerts, unusual data transfers

Mapping Detections to Controls

Detection Rule with Compliance Mapping 
id: detect-privilege-escalation
name: Privileged Role Assignment Detection
description: Detect when users are assigned high-privilege roles

compliance_mapping:
  nist_csf:
    - PR.AC-4  # Access permissions managed
    - DE.CM-3  # Personnel activity monitored
  nist_800_53:
    - AC-2     # Account Management
    - AC-6     # Least Privilege
    - AU-6     # Audit Review, Analysis
  iso_27001:
    - A.9.2.3  # Privileged access rights
    - A.12.4.1 # Event logging
  pci_dss:
    - 7.1      # Limit access to system components
    - 10.2.5   # Use of identification and authentication
  cis_controls:
    - 5.4      # Restrict Administrator Privileges
    - 8.5      # Collect Detailed Audit Logs
  mitre_attack:
    - T1078    # Valid Accounts
    - T1098    # Account Manipulation

detection:
  platform: sentinel
  query: |
    AuditLogs
    | where OperationName has "Add member to role"
    | where TargetResources[0].modifiedProperties[0].newValue has_any 
        ("Global Administrator", "Security Administrator", "Privileged Role Administrator")
    | project TimeGenerated, InitiatedBy, TargetUser, Role

Interview Preparation

Key points:

  • Map detections to control requirements (NIST, ISO, PCI)
  • Ensure detection rules provide audit evidence
  • Use frameworks to prioritize detection development
  • Generate compliance reports from SIEM data
  • Work with GRC team to understand requirements

Example: "I maintain a mapping of our Sentinel analytics rules to NIST 800-53 controls. This helps us identify coverage gaps and provides evidence for AU-6 (Audit Review) during assessments."

NIST CSF:

  • High-level risk management framework
  • 5 functions: Identify, Protect, Detect, Respond, Recover
  • Flexible, outcome-focused
  • Good for executive communication

NIST 800-53:

  • Detailed control catalog (800+ controls)
  • 20 control families (AC, AU, CM, etc.)
  • Prescriptive implementation guidance
  • Required for federal systems (FISMA)

Relationship: CSF provides the "what" (outcomes), 800-53 provides the "how" (specific controls). CSF references 800-53 controls.
